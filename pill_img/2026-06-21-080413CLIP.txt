● 네, 맥락 정리해서 설명드리겠습니다.

  1. PARSeq — 각인 문자 인식(STR) 모델

  YOLO가 알약에서 "각인이 있는 영역"을 박스로 찾아내면, 그 박스를 crop한 이미지
  안의 글자가 무엇인지 읽어내는 모델이 필요합니다. 이게 PARSeq입니다.

  - Scene Text Recognition(STR) 분야의 사전학습 모델 (Transformer 기반,
  ECCV 2022)
  - 처음엔 우리 데이터(862개)로 CRNN을 처음부터 학습시켰는데 실패했습니다 —
  정확도 0.7%. 데이터가 너무 적어서(고유 텍스트 530개, 평균 1.6번씩만 등장)
  처음부터 배우는 모델은 일반화가 안 됐습니다.
  - 그래서 이미 수백만 장의 일반 장면 텍스트(거리 표지판, 간판 등)로 학습된 
  PARSeq를 가져와서, 우리의 적은 데이터(862개)로 **미세조정(fine-tuning)**만
  했습니다. 이미 "글자를 읽는 법" 자체는 알고 있으니, "알약 각인 도메인"에만
  적응시키는 셈이라 적은 데이터로도 잘 됨 — 정확도가 48~62%까지 올라갔습니다.

  이게 바로 transfer learning의 전형적인 사례입니다. 사전지식 없이 처음부터
  배우기엔 데이터가 부족했지만, 이미 잘 아는 모델을 살짝 조정하는 건 적은
  데이터로도 충분했던 것이죠.

  2. Lexicon 보정 — 후처리 단계

  PARSeq가 글자를 읽어도 완벽하진 않습니다(예: "9109"라고 잘못 읽음, 실제는
  "3109"). 그래서 **약물사전(우리가 Gemini로 뽑은 530개 고유 각인 텍스트 목록,
  drug_imprints.txt)**을 만들어두고, PARSeq의 예측 결과를 이 사전에서 가장 
  가까운(편집거리 2 이내) 후보로 교정합니다.

  "9109" → 사전에 "3109"가 있고 편집거리 1 → "3109"로 교정

  이건 closed-set(약물사전 안에 정답이 있다는 가정) 보정이라, 사전 밖의 신약
  각인은 못 고쳐줍니다 — 사전을 계속 키워나가야 하는 한계가 있습니다.

  3. ArcFace — 외형(색/모양) 임베딩 학습

  각인 텍스트만으로는 알약을 완전히 식별할 수 없어서(같은 각인, 다른 모양/색의
  약이 있을 수 있음), 알약 사진을 보고 "이게 어떤 약과 닮았나"를 판단하는 별도 
  모델이 필요했습니다.

  문제는 약물당 사진이 1장(앞)+1장(뒤)뿐이라는 것 — 일반 분류기처럼 "이 클래스는
  사진 100장으로 학습"이 안 됩니다. ArcFace는 원래 얼굴인식에서 쓰는 기법으로,
  "같은 사람의 다른 사진들은 임베딩 공간에서 가깝게, 다른 사람은 멀게"
  배치하도록 학습시킵니다. 우리는 사진이 1장뿐이라, **매 epoch마다
  회전/색상변화/크롭 등 강한 augmentation을 걸어서 "가짜 멀티뷰"**를 만들어 같은
  약 사진들끼리는 가깝게, 다른 약과는 멀게 배우도록 했습니다.

  4. FAISS — 네, 맞습니다, 같은 용도입니다

  LLM에서 vector DB로 쓰던 것과 본질적으로 똑같은 용도입니다. 구조가 동일합니다:

  ┌────────────┬────────────────────────────┬───────────────────────────────┐
  │            │          LLM RAG           │             여기              │
  ├────────────┼────────────────────────────┼───────────────────────────────┤
  │ 임베딩     │ 텍스트 → embedding 모델 →  │ 알약 사진 → ArcFace 모델 →    │
  │ 생성       │ 벡터                       │ 벡터                          │
  ├────────────┼────────────────────────────┼───────────────────────────────┤
  │ 저장       │ FAISS 인덱스               │ FAISS 인덱스                  │
  ├────────────┼────────────────────────────┼───────────────────────────────┤
  │ 검색       │ 질문 벡터로 최근접 문서    │ 새 알약 사진 벡터로 최근접 약 │
  │            │ 찾기                       │  찾기                         │
  └────────────┴────────────────────────────┴───────────────────────────────┘

  차이는 임베딩을 만드는 모델이 텍스트 임베딩 모델이 아니라 우리가 직접 학습시킨
  이미지 임베딩 모델(ArcFace)이라는 것뿐입니다. 지금은 413개 약(앞+뒤, 826개
  벡터)을 인덱스에 넣어놓고, 새 알약 사진이 들어오면 그 사진의 벡터로 "가장
  가까운 벡터 5개"를 찾아서 어떤 약일 가능성이 높은지 보는 식입니다(top-5 92%
  정확도). RAG에서 "이 질문과 가장 관련된 문서 5개 찾기"랑 정확히 같은
  원리입니다 — 새 약이 추가돼도 모델을 재학습할 필요 없이 인덱스에 벡터만 
  추가하면 되는 것도 RAG의 "새 문서 추가" 패턴과 동일합니다.

● ArcFace 학습(40 epoch) 상세 설명

  무엇을 했는지:
  - 알약은 클래스(약)당 진짜 사진이 앞면 1장, 뒷면 1장뿐이라 일반 분류기 학습이
  안 됨
  - 그래서 매 epoch마다 같은 사진에 강한 augmentation(회전 180도, 색상변화, 
  랜덤크롭 등)을 다르게 걸어서 "가짜 멀티뷰"를 만들고, views_per_class=10으로 한
  클래스당 epoch마다 10개의 변형판을 보여줌 (352클래스 × 10 = 3,520장/epoch,
  batch=32)
  - backbone은 처음엔 efficientnetv2_rw_s를 썼다가 GTX1050(2GB)에서 너무
  무거워서(20분에 1 epoch도 못 끝냄) **mobilenetv3_small_100**으로 교체 —
  가벼워서 epoch당 1~2분으로 단축됨
  - ArcFace margin loss로 "같은 클래스(같은 약의 augmentation 변형들)는 임베딩
  공간에서 가깝게, 다른 약은 멀게" 배치하도록 학습

  40 epoch 동안의 추이 (로그 기준):
  ep1: loss=13.75 acc=0%
  ep10: acc=41%
  ep20: acc=67%
  ep30: acc=79%
  ep34: acc=83.7% (최고)
  ep35~40: acc 82~84% 사이에서 오르락내리락 (정체)

  Epoch을 늘리면 더 좋아질까?

  짧게 답하면: 거의 효과 없을 가능성이 높고, 오히려 손해일 수도 있습니다.

  이유:
  1. train_acc 곡선이 ep34쯔음부터 명백히 평탄해졌습니다 — 0%→80%대까지는 빠르게
  올랐는데, 마지막 6 epoch은 82~84% 사이를 맴돌 뿐 더 안 오릅니다. 이건 "아직
  배우는 중"이 아니라 "이 설정으로 도달 가능한 한계에 거의 다 왔다"는
  신호입니다.
  2. 더 중요한 문제: 제가 추적한 train_acc는 학습에 쓰인 352개 클래스에 대한
  분류 정확도일 뿐이고, 진짜 중요한 지표(미학습 61개 클래스의 검색 정확도, 
  top5=91.8%)는 학습 끝나고 딱 한 번만 측정했습니다. epoch을 늘렸을 때 이 val
  지표가 실제로 좋아지는지 나빠지는지는 데이터가 없어서 모릅니다.
  3. 오히려 위험한 방향: 클래스당 진짜 이미지가 1장뿐이라, epoch을 너무 늘리면
  모델이 "그 사진의 augmentation 패턴 자체"를 외워버리고(과적합) 진짜 일반화
  가능한 시각적 특징을 덜 배우게 될 수 있습니다. 이러면 train_acc는 비슷하게
  유지되거나 더 올라도, val 검색 정확도는 떨어질 수 있습니다.

  그래서 추천하는 건 "epoch을 무작정 늘리기"가 아니라:
  - 5~10 epoch마다 체크포인트를 저장하고 매번 eval_retrieval.py로 val
  top1/top5를 찍어서, val 정확도가 최고인 epoch을 찾는 방식(early stopping
  개념)이 훨씬 합리적입니다.
  - 또는 데이터 자체를 늘리는 게(E: 드라이브 마운트해서 4392개로) epoch
  늘리기보다 효과가 클 가능성이 높습니다 — 클래스 수가 늘면 ArcFace가 "다른 약과
  구분되는 특징"을 더 풍부하게 배울 기회가 생기기 때문입니다.

  원하시면 체크포인트별 val 추적하는 학습 스크립트로 고쳐서 실제로 "몇 epoch이
  최적인지" 확인해볼까요?

  1장)**일 가능성이 큽니다. 큰 backbone은 파라미터가 많아서 오히려 이 적은
  데이터에 더 쉽게 과적합할 위험도 있습니다.

  그래서 추천: backbone을 키우는 것보다, 먼저 (1) val 추적으로 최적 epoch을 찾고
  (2) E: 드라이브 마운트해서 데이터를 늘리는 게 투자 대비 효과가 클 것
  같습니다. 정말 궁금하시면 efficientnetv2_rw_t(tiny 버전, _s보다 훨씬 가벼움)
  정도가 절충안이 될 수 있는데, 이건 데이터 늘린 후에 시도해보는 걸 추천합니다.

  ## 왜 단일 방법이 아니라 cascade인가                                          
                                                                                
  1번(이미지 특징 휴리스틱)은 빠르고 공짜지만 경계 케이스에서 새고, 2번(Gemini  
  2차 검증)은 정확하지만 749개×896페이지를 또 호출하면 비용·시간이 큽니다. 둘의 
  장점만 취하려면 단계로 거는 게 맞습니다.                                      
                                                                                
  ```                                                                           
  749개 crop                                                                    
    → [1단계 무료 휴리스틱 필터]                                                
          ├─ 명백한 알약    → 통과 (재호출 없음)                                
          ├─ 명백한 텍스트  → 폐기 (재호출 없음)                                
          └─ 애매한 것만    → [2단계 Gemini 검증]  ← 전체의 일부만              
  ```                                                                           
                                                                                
  휴리스틱이 양 극단(확실한 알약/확실한 텍스트)을 처리하고, 점수가 중간인       
  회색지대만 Gemini로 넘기면 — Gemini 호출이 전체의 10~20%로 줄어 비용은 2번의  
  일부인데 정확도는 거의 2번 수준이 됩니다. 이게 우리가 앞서 얘기한 cascade     
  원리와 같습니다.                                                              
                                                                                
  ## 1단계 무료 휴리스틱 — "텍스트 vs 알약"을 가르는 신호                       
                                                                                
  말씀하신 텍스트/엣지 밀도가 핵심이 맞고, 알약 사진과 텍스트 패치는 시각       
  통계가 꽤 다릅니다. 효과적인 신호들:                                          
                                                                                
  **엣지/텍스트 밀도.** 텍스트는 잔 엣지가 격자처럼 고르게 퍼져 있고, 알약은    
  중앙에 하나의 큰 매끈한 덩어리 + 배경입니다. Canny 엣지 비율, 또는            
  MSER/획(stroke) 영역 비율이 텍스트에서 훨씬 높습니다.                         
                                                                                
  **연결요소(connected component) 구조.** 텍스트는 비슷한 크기의 작은           
  컴포넌트가 여러 개 줄지어 있고, 알약은 큰 컴포넌트 1~2개입니다. "큰 컴포넌트  
  1개가 프레임의 X% 이상"이면 알약 쪽.                                          
                                                                                
  **색/채도.** 텍스트 카드는 보통 흰 배경+검은 글자(저채도·고대비 이분          
  히스토그램)인데, 알약은 색이 있고(채도↑) 명도 분포가 매끈합니다. HSV 채도     
  평균·분산이 좋은 분리 신호입니다.                                             
                                                                                
  **원형성/형태.** 알약은 원·타원·캡슐형 → 최대 컨투어의                        
  원형도(circularity)·종횡비가 일정 범위. 텍스트 블록은 직사각형 가득 참.       
                                                                                
  이 신호들을 합쳐 점수를 내고, 명백/회색지대로 3분할하면 됩니다. 다만          
  **임계값은 반드시 데이터로 보정**하세요 — 한국 알약식별 페이지의 텍스트 카드  
  스타일에 맞춰야 합니다.                                                       
                                                                                
  ## 하지만 더 나은 방법 — "필터"를 "분류기"로                                  
                                                                                
  휴리스틱은 임계값 튜닝이 까다롭고 새는 케이스가 남습니다. 선생님은 이미       
  30개를 라벨링(알약/텍스트)했고 contact sheet로 더 만들 수 있으니, **가벼운    
  2-class 분류기(알약 vs 비알약)를 학습**하는 게 휴리스틱보다 견고합니다.       
                                                                                
  가장 빠른 길은 **임베딩 + 분류**입니다. 사전학습 MobileNet/CLIP으로 각 crop을 
  임베딩한 뒤, 수백 개 라벨로 로지스틱 회귀나 작은 MLP를 얹으면 됩니다.         
  휴리스틱보다 일반화가 좋고, 학습/추론이 공짜에 가깝습니다. CLIP을 쓰면        
  **zero-shot**으로 "a photo of a pill" vs "a page of text" 프롬프트            
  유사도만으로도 라벨 없이 1차 분리가 가능해서, 이걸 1단계로 두고 점수 중간만   
  Gemini로 넘기는 구성이 가장 실용적입니다.                                     
                                                                                
  이 분류기 방식의 부수 효과: 나중에                                            
  page_type(photo_card/code_index_table/other)도 같은 임베딩으로 분류할 수      
  있어, 추출 파이프라인 전체가 일관된 구조가 됩니다.                            
                                                                                
  ## 가장 근본적 — 추출 단계 자체를 바꾸기                                      
                                                                                
  지금 "검출율이 낮다"의 상당 부분은 **Gemini에게 crop 좌표를 직접 시키는       
  방식의 한계**입니다(앞서 본 MLLM의 느슨한 박스·환각). 페이지가 photo_card 69% 
  / code_index_table 18%로 **정형화된 레이아웃**이라면, 알약 사진은 격자 셀에   
  규칙적으로 배치돼 있을 가능성이 큽니다. 그렇다면:                             
                                                                                
  - **레이아웃 기반 crop**: 페이지를 격자 셀로 분할하는 규칙(또는 가벼운 셀

  분류만 정확해도 쓰레기의 큰 덩어리가 사라집니다.                              
                                                                                
  즉 "각 crop이 알약인지 사후 판별"하기 전에, "**페이지 유형을 먼저 분류해      
  텍스트 페이지는 추출조차 안 하는**" 게 상류에서 오염을 막는 더 깨끗한         
  방법입니다.
